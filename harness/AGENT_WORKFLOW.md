# Agent Workflow & Action Guidelines

## 1. 작업 프로세스 (절대 규칙)

- 코드 수정 전 현재 브랜치 루트에 `PLAN.md`를 생성한다.
- **`PLAN.md` 필수 포함 내용:**
  1. 요구사항 분석 및 작업 범위
  2. 생성/수정할 파일 목록 (Mock 파일 포함)
  3. **테스트 시나리오:** `harness/TESTING-PROTOCOL.md`를 참고하여 단계별 테스트 계획(Phased Test Plan)을 포함할 것.
- 계획 작성이 완료되면 사용자에게 승인을 요청하며, **명시적인 승인(Approve) 전까지는 절대 코드를 작성하지 않는다.**
- 승인 후 `agent/기능-이름` 브랜치를 생성하여 이동한 뒤 작업을 수행한다.

## 2. 코딩 컨벤션

- 네이밍 : 변수 및 함수명은 비즈니스 의도가 잘 드러나는 영어 시맨틱 네이밍을 준수하며, 가급적 임의의 축약어 사용은 피한다.
- 주석 작성: 가급적이면 네이밍으로 코드의 의도를 드러내는 데 주력하고, 이모티콘(Emoji)을 배제한 순수 텍스트로만 작성한다.
- 관심사의 분리(Separation of Concerns): UI 렌더링 로직과 비즈니스 로직을 철저히 분리한다. 복잡한 데이터 가공이나 상태 관리 로직은 Custom Hook(예: `useGraphData.ts` 패턴)으로 추출하고, UI 컴포넌트는 화면을 그리는 데만 집중하도록 설계한다.

## 3. Mock 데이터 및 파일 생성 (Mocking First)

- 사용자의 명시적 지시가 없는 한 실제 API(`springClient`)를 호출하지 않는다.
- 에이전트는 제공된 DTO 구조를 바탕으로 필요한 Mock 데이터를 스스로 설계하고 별도의 파일(예: `Label.mock.ts`)로 생성해야 한다.
- Mock 데이터는 한국어 기반의 실감 나는 예시 데이터로 구성한다.
